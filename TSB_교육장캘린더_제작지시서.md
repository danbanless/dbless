## 2. 데이터 소스: 예약 시스템 API 명세

베이스 URL: `https://hanwhahrdvenue-production.up.railway.app`
아래 3개 엔드포인트 모두 **인증 불필요(공개)**, 응답은 JSON.

### 2-1. `GET /api/schedule?year=2026&month=7` — 월별 예약 일정 (핵심)

- 취소(`cancelled`) 예약은 서버가 이미 제외하고 반환한다.
- 여러 날짜에 걸친 예약은 **날짜별로 펼쳐진 배열**로 온다 (1예약 1날짜 1행).
- `year`/`month` 생략 시 전체 예약 반환 — 반드시 파라미터를 붙여 호출할 것.

응답 예시:
```json
[
  {
    "id": "mqm0skks7fowx",
    "date": "2026-06-22",
    "startTime": "08:30",
    "endTime": "17:30",
    "rooms": [1],
    "combined": false,
    "status": "confirmed",
    "courseName": "AI교육"
  }
]
```

필드 정의:

| 필드 | 타입 | 의미 |
|---|---|---|
| `id` | string | 예약 ID. 같은 예약이 여러 날짜로 펼쳐지면 id가 중복됨 |
| `date` | string `YYYY-MM-DD` | 해당 일자 |
| `startTime` / `endTime` | string `HH:mm` | 시작/종료 시각 |
| `rooms` | number[] | 사용 교육장. `[1]`, `[2]`, `[1,2]` 중 하나 |
| `combined` | boolean | `true`면 합실(1·2교육장을 터서 하나로 사용) |
| `status` | string | `confirmed`(확정) 또는 `pending`(승인 대기) |
| `courseName` | string | 과정명 (캘린더 칩 제목으로 사용) |

### 2-2. `GET /api/blocked-dates` — 대관 불가일

응답: 날짜 문자열 배열. 예: `["2026-07-15", "2026-08-01"]`
→ 캘린더에서 해당 날짜를 회색 처리하고 "대관 불가" 배지 표시.

### 2-3. `GET /api/holidays?year=2026&month=7` — 공휴일

서버가 한국천문연구원 API를 프록시해준다. **위젯에 별도 API 키가 필요 없다.**
`month` 생략 시 연간 전체 반환 — 연 1회 연간 조회 후 캐시하는 방식 권장.

응답 예시:
```json
[{ "date": "2026-08-15", "name": "광복절" }]
```

## 4. 화면 설계

### 4-1. 전체 레이아웃

TSB의 기존 위젯 룩앤필(테두리 없는 카드형, 라이트/다크, 투명도 슬라이더)을 유지하되 내부는 단순화:

```
┌──────────────────────────────────────────────┐
│ 서울 교육장 일정   [일간|주간|월간] ◀ 오늘 ▶  🌙 🔅 ⟳ │  ← 헤더
├──────────────────────────────────────────────┤
│                                              │
│   (뷰 영역: 일간/주간/월간 캘린더)             │
│                                              │
├──────────────────────────────────────────────┤
│ 마지막 갱신 14:32  ·  ● 정상            [범례] │  ← 상태바
└──────────────────────────────────────────────┘
```

- 헤더: 타이틀, 뷰 전환(일간/주간/월간), 날짜 이동(◀ 오늘 ▶), 다크모드 토글, 투명도 슬라이더, 수동 새로고침(⟳)
- 상태바: 마지막 갱신 시각 + 연결 상태(§6), 범례(확정/대기/합실/대관불가/공휴일)

### 4-2. 레인 구조 — 핵심 규칙

TSB의 "팀장 레인" 개념을 **"교육장 레인"**으로 치환한다. 일간/주간 뷰에서:

```
        │ 09:00      12:00      15:00      18:00
────────┼───────────────────────────────────────
1교육장 │   [AI교육 (확정)      ]
2교육장 │        [리더십과정 (대기)]
```

- 레인은 항상 2개 고정: **1교육장**, **2교육장**
- `rooms:[1]` → 1교육장 레인, `rooms:[2]` → 2교육장 레인, `rooms:[1,2]` → 양쪽 레인 각각 표시
- **`combined: true`** → 하나의 칩이 두 레인을 세로로 관통하는 형태로 표시하고 "합실" 배지 부착

### 4-3. 뷰 3종

1. **일간 뷰**: 가로 시간축(08:00~19:00) × 세로 2레인. 칩에 과정명 + 시간 표시.
2. **주간 뷰** (기본 뷰): 가로 7일(월~일) × 각 날짜 안에 1·2교육장 행. 칩에 과정명만.
3. **월간 뷰**: 표준 달력 그리드. 각 날짜 셀에 예약 칩을 최대 3개까지 표시, 초과분은 "+N". 날짜 셀 클릭 시 그 날짜의 일간 뷰로 전환.

### 4-4. 색상 규칙

DBLess 공통 디자인 시스템(`dbless-common.css` 토큰)의 색상 체계를 따른다:

| 대상 | 스타일 |
|---|---|
| `confirmed` | 테라코타(교육 색상) 배경, 흰 글자 |
| `pending` | 같은 색상 계열의 **연한 톤 + 점선 테두리** + "대기" 배지 |
| `combined` 합실 | 확정/대기 색 규칙 그대로 + 두 레인 관통 + "합실" 배지 |
| 대관 불가일 | 날짜 셀 회색 처리 + "대관 불가" 표시 |
| 공휴일 | 날짜 숫자 빨간색 + 공휴일명 소문자 표기 |
| 오늘 | 날짜 셀 인디고 하이라이트 테두리 |

### 4-5. 칩 클릭 동작

읽기 전용이므로 수정 UI는 없다. 칩 클릭 시 **상세 팝오버**만 표시:
과정명, 날짜, 시간, 교육장(1/2/합실), 상태(확정/대기). 닫기 버튼 외 액션 없음.

## 6. 폴링·오류 처리 규칙

### 6-1. 폴링 사이클

1. 앱 시작 즉시 1회 호출, 이후 `DispatcherTimer`로 **5분마다** 반복.
2. 매 사이클 호출 대상:
   - `/api/schedule` — **현재 보고 있는 달 + 다음 달** 2회 호출 (월말에 다음 달 예약이 안 보이는 문제 방지)
   - `/api/blocked-dates`
3. `/api/holidays?year=YYYY`는 **연 단위 1회만** 호출하고 메모리 캐시 (연도가 바뀌거나 뷰가 다른 연도로 이동할 때만 추가 호출).
4. 수동 새로고침(⟳) 버튼은 즉시 1사이클 실행.
5. 사용자가 ◀▶로 다른 달로 이동하면 그 달 데이터가 캐시에 없을 때만 즉시 호출.

### 6-2. 실패 처리 — "마지막 데이터 유지" 원칙

- HTTP 실패/타임아웃(10초) 시: **화면 데이터를 지우지 않는다.** 마지막 성공 데이터를 그대로 유지.
- 상태바 표시 규칙:
  - 정상: `● 정상 · 마지막 갱신 HH:mm` (초록)
  - 1회 이상 연속 실패: `● 갱신 실패 · 마지막 성공 HH:mm` (주황) — 팝업/알림창 띄우지 말 것
- 실패해도 다음 5분 주기에 자동 재시도. 별도 백오프 로직 불필요.
- 앱 시작 시점부터 서버에 닿지 못하면: 빈 캘린더 + 상태바에 `서버에 연결할 수 없습니다` 표시.

### 6-3. 데이터 병합 규칙

- 매 폴링마다 해당 월 데이터를 **전체 교체(replace)** 한다. diff/merge 불필요 — 읽기 전용이므로 로컬 변경분이 없다.
- 여러 날짜에 걸친 예약은 API가 이미 날짜별 행으로 펼쳐서 주므로 추가 가공 불필요. 단, 같은 `id`가 여러 행에 나타날 수 있음을 상세 팝오버에서 유의 (그 행의 `date` 기준으로 표시하면 됨).

## 9. 개발 순서 (권장 마일스톤)

1. **M1 — 프로젝트 뼈대**: csproj + App + 빈 MainWindow. DesktopPin/AppSettings 복사·적용. 실행 시 바탕화면 고정 위젯 창이 뜨는 것 확인.
2. **M2 — API 서비스**: `VenueApiService` 작성. 콘솔/디버그 출력으로 3개 API 응답 파싱 확인. (Railway 실서버 대상 — 테스트 데이터가 이미 있음: 2026-06월 예약)
3. **M3 — 주간 뷰**: 2레인 주간 캘린더 렌더링 + 색상 규칙 + 합실 관통 표시. 여기까지가 MVP.
4. **M4 — 일간/월간 뷰** + 뷰 전환/날짜 이동.
5. **M5 — 마감**: 폴링 타이머, 실패 처리 상태바, 상세 팝오버, 공휴일/대관불가 표시, 트레이 아이콘, 다크모드.
6. **M6 — 배포**: single-file publish → `배포\VenueCalendar.exe` 생성, 실PC에서 Win+D·재시작·사외망(실패 처리) 시나리오 수동 검증.
